## 1. Typed Edges

### How does Sim store edge metadata?

Edges are stored as **React Flow `Edge[]`** in `WorkflowState.edges`
(`apps/sim/stores/workflows/workflow/types.ts`). Each edge is a standard
React Flow edge with these fields used by Sim:

```typescript
{
  id: string               // UUID (via generateId())
  source: string           // source block ID
  target: string           // target block ID
  sourceHandle?: string    // handle name on source block
  targetHandle?: string    // handle name on target block
  type?: string            // React Flow edge type (default: 'default')
  data?: Record<string, any>  // arbitrary payload — see below
}
```

In the database (`packages/db/schema.ts`, `workflowEdges` table):
- Columns: `id`, `workflowId`, `sourceBlockId`, `targetBlockId`, `sourceHandle`, `targetHandle`
- **No `data` / `metadata` column exists at the DB level.**

Edge `data` is populated at render time in `workflow.tsx` (~line 3869):
```typescript
data: {
  ...edge.data,         // spread any existing data
  isSelected: ...,
  isInsideLoop: ...,
  sourceHandle: ...,
  onDelete: ...,
}
```

### Can edges carry custom data (like `relationship: 'DEPENDS_ON' | 'APPEALS'`)?

**In memory: yes.** The React Flow `Edge.data` field is untyped and accepts any object.
You can store `data: { relationship: 'DEPENDS_ON' }` when creating an edge via
`batchAddEdges()` in the workflow store.

**At the database layer: no persistence today.** The `workflowEdges` table has no
`data` or `metadata` column. Custom data set in memory is lost on save/reload unless
you either:
- **Option A:** Add a `metadata jsonb` column to `workflowEdges` via a DB migration.
- **Option B:** Encode the relationship type in the `sourceHandle` name (e.g.,
  `source-depends_on`, `source-appeals`), which *is* persisted and is already used by
  condition/router to select paths.

**For Phase A, Option B (handle-encoded relationship) is the zero-migration path.** It also
slots cleanly into the edge-manager's `shouldActivateEdge()` logic.

### How are edge types rendered visually?

`WorkflowEdge` (`components/workflow-edge/workflow-edge.tsx`) renders a SmoothStep SVG
path. Visual styling is determined by:

1. **Diff status** (`deleted` → dashed red, `new` → brand-accent colour)
2. **Run status** (`success` → green, `error` → red)
3. **Error handle** (`sourceHandle === 'error'` → always red)
4. **Selection** → opacity 0.5, width 2.5px
5. **Default** → `var(--workflow-edge)` colour, solid, 2px

There is **no edge `type` dispatch** to different renderer components — one renderer
handles all edges. Visual variation is driven by the `data` fields and handle names
read at render time, not by a registered `edgeTypes` map.

### Can different edge types have different visual styles?

**Not via the current `edgeTypes` map** (only `'default'` / `'workflowEdge'` is
registered). But since `WorkflowEdge` reads `edge.data` freely, you can add any flag
you like:

```typescript
// When creating governance edges:
batchAddEdges([{
  id: generateId(),
  source: 'blockA',
  target: 'blockB',
  data: { governanceRelationship: 'DEPENDS_ON' },
}])
```

Then in `workflow-edge.tsx` add:
```typescript
const isGovernanceEdge = data?.governanceRelationship
if (isGovernanceEdge === 'APPEALS') {
  color = '#f59e0b'
  style.strokeDasharray = '6,3'
}
```

This requires no new edge type registration — just extending the one renderer.
**For persistence, add a DB migration for `metadata jsonb` on `workflow_edges`.**

### What would it take to add a governance edge type?

Minimal path:
1. **In-memory styling only:** Extend `WorkflowEdge` renderer to read
   `data.governanceRelationship` and branch on colours/dash patterns. (~10 LOC)
2. **Persisted relationship type:** Add `metadata jsonb` column to `workflowEdges`
   table (one migration). Populate it in the serializer/copilot operations.
3. **Alternatively: handle-encoded type** — encode relationship in `sourceHandle`
   (e.g., `source-depends_on`). Already persisted. Edge manager may need a passthrough
   case in `shouldActivateEdge()` for governance-specific handles.

**Phase A recommendation:** Use handle-encoding for relationship type (zero DB changes);
add a visual style hint in `WorkflowEdge` based on handle prefix. Ship DB migration in
Phase B.

## 2. Block Input/Output Type System

### How are block inputs and outputs typed?

**Inputs** (`BlockConfig.inputs`) are `Record<string, ParamConfig>`:
```typescript
interface ParamConfig {
  type: 'string' | 'number' | 'boolean' | 'json' | 'array' | 'file'
  description?: string
  schema?: { ... }   // JSON Schema for json/array types
}
```
File: `apps/sim/blocks/types.ts`

**Outputs** (`BlockConfig.outputs`) are `Record<string, OutputFieldDefinition>`:
```typescript
type OutputFieldDefinition =
  | PrimitiveValueType                    // 'string' | 'number' | ... (shorthand)
  | {
      type: PrimitiveValueType
      description?: string
      condition?: OutputCondition         // show output only under certain param values
      hiddenFromDisplay?: boolean
    }
```

The serializer (`apps/sim/serializer/types.ts`) copies `inputs` and `outputs` into
`SerializedBlock.inputs` / `SerializedBlock.outputs`.

### Is there type validation when block A's output connects to block B's input?

**No runtime type validation at connection time.** React Flow allows connecting any
source handle to any target handle. The system does validate:
- **Cycle detection** — `wouldCreateCycle()` prevents circular edges.
- **Scope validation** — `removeInvalidScopeEdges()` drops cross-scope (loop/parallel)
  edges.
- **Copilot validation** — `validateInputsForBlock()` in the copilot tool checks types
  before applying AI-generated changes.

At execution time the resolver (`executor/variables/resolver.ts`) resolves
`<blockName.outputField>` references; mismatches produce runtime errors, not compile-time
ones.

**Implication for governance:** No automatic "evidence_in only accepts json" guard. You
can document the expected type in `ParamConfig.description` and enforce at the handler level.

### Can a block have multiple named input handles?

**No — not natively.** Every block renders exactly **one** target handle with id
`'target'` (left or top depending on `horizontalHandles`). This is hardcoded in
`workflow-block.tsx`:

```tsx
<Handle type='target' id='target' ... isConnectableEnd={true} />
```

There is no provision for additional target handles. Multiple inputs are instead modelled
as **sub-block fields** (the user fills them in the panel editor) that reference upstream
block outputs via the `<blockName.field>` interpolation syntax.

**To add multiple named input handles (e.g., `evidence_in`, `authority_in`):**
- Register additional `<Handle type='target' id='evidence_in' .../>` components in
  `workflow-block.tsx` conditioned on `type === 'governance_gate'` (same pattern as
  the dynamic output handles for condition/router).
- Update the serializer to preserve the `targetHandle` for these edges.
- Update `EdgeConstructor` in `executor/dag/construction/edges.ts` to wire them into
  `DAGNode.incomingEdges`.
- Update the handler to pull named inputs from resolved `inputs` map.

This is a **medium effort** change (~100–200 LOC) but it is fully supported by the
React Flow architecture.

### How are handles rendered?

Handles are `<Handle>` components from `reactflow`. Positioning:
- Default: `type='target'` at top/left (based on `horizontalHandles`), `type='source'`
  at bottom/right.
- Condition rows: dynamic source handles at `top: CONDITION_START_Y + n * ROW_HEIGHT`
  (per row, computed from the `conditionRows` array).
- Router v2 routes: same pattern, offset by one row for the context field.
- Error handle: fixed offset from the bottom-right.

Handles are **position-absolute on the node**, sized via CSS, and registered with React
Flow's `updateNodeInternals` when count changes
(`hooks/use-dynamic-handle-refresh.ts`).

### How do condition and router blocks work with multiple outputs?

**Condition block** (`blocks/blocks/condition.ts`, handler:
`executor/handlers/condition/condition-handler.ts`):
- `condition-input` sub-block defines N rows (if / else-if / else).
- Each row gets a source handle: `condition-<rowId>`.
- Handler evaluates JS expressions, returns `selectedOption: string` (the matching row
  ID).
- `EdgeManager.shouldActivateEdge()` matches `handle.startsWith('condition-')` →
  activates only the edge whose handle suffix equals `selectedOption`.

**Router v2** (`blocks/blocks/router.ts`, handler:
`executor/handlers/router/router-handler.ts`):
- `router-input` sub-block defines N route descriptions.
- Each route gets a source handle: `router-<routeId>`.
- Handler sends routes + context to an LLM, returns `selectedRoute: string`.
- `EdgeManager.shouldActivateEdge()` matches `handle.startsWith('router-')`.

**Key insight:** Any block can have dynamic outputs by:
1. Defining dynamic rows in a sub-block (like `condition-input` or `router-input`).
2. Rendering one source handle per row in `workflow-block.tsx`.
3. Writing a handler that returns the chosen branch in `selectedOption` or
   `selectedRoute`.

## 3. Block Categories

### How does the `category` field affect rendering, menu placement, search?

`BlockCategory = 'blocks' | 'tools' | 'triggers'`
File: `apps/sim/blocks/types.ts`

In the **Toolbar** (`components/panel/components/toolbar/toolbar.tsx`):
```typescript
const regularBlockConfigs = allBlocks.filter(b => b.category === 'blocks')
const toolConfigs         = allBlocks.filter(b => b.category === 'tools')
```
- `'blocks'` → appears in the "Blocks" section of the toolbar (top area), sorted
  alphabetically alongside Loop and Parallel synthetic entries.
- `'tools'` → appears in the "Blocks" section below the core blocks, also sorted
  alphabetically.
- `'triggers'` → appears in the "Triggers" section (separate from blocks entirely).

The ⌘K search modal uses the same `getBlocksForSidebar()` function and therefore the
same filtering logic.

`integrationType` (`IntegrationType` enum) and `tags` (`IntegrationTag[]`) exist on
`BlockConfig` but are **not currently used for grouping in the toolbar**. The toolbar
does not group blocks by integration type — it puts all `'blocks'` together and all
`'tools'` together, sorted alphabetically.

### Can we add a new category without breaking anything?

**Technically yes, but it requires UI changes.** The `BlockCategory` type is a union
`'blocks' | 'tools' | 'triggers'`. Adding `'governance'` requires:
1. Extend the union in `apps/sim/blocks/types.ts`.
2. Update `getBlocksForSidebar()` in `lib/workflows/triggers/trigger-utils.ts` to
   include the new category (currently it only excludes `'triggers'`).
3. Update the Toolbar component to render a new section for `'governance'` blocks.
4. Update any other `category === 'blocks'` guards.

**Simpler alternative: use `category: 'blocks'` with a new `IntegrationTag` value
`'governance'`.** The tag is already part of `BlockConfig` and nothing breaks if you
add a new string to the `IntegrationTag` union — the toolbar simply doesn't use it for
grouping today. This is the zero-friction path.

### Is the category list hardcoded somewhere in the UI?

The `BlockCategory` type is declared once in `apps/sim/blocks/types.ts`. The toolbar
uses it via `block.category === 'blocks'` and `block.category === 'triggers'` filter
expressions — no switch/case or exhaustive enum. **Adding a new value without touching
those two filter expressions means the new category is invisible in the toolbar.**

### Is there a `table` type that supports dynamic rows with multiple columns?

**Yes.** `type: 'table'` is a full editable grid. Configuration:
```typescript
{
  id: 'voterList',
  type: 'table',
  title: 'Voter List',
  columns: ['Name', 'Role', 'Weight'],   // column headers
  defaultValue: [],                       // initial rows (array)
}
```
The value stored is an array of arrays (`string[][]`). The component
(`sub-block/components/table/`) renders a dynamic row table with add/remove row
controls. This is directly suitable for voter lists, evidence tables, and delegation maps.

From the `SubBlockState` type:
```typescript
interface SubBlockState {
  value: string | number | string[][] | null
}
```
The `string[][]` format confirms multi-column row storage.

### Is there a way to have conditional sub-blocks (show field X only if field Y = Z)?

**Yes — fully supported.** `SubBlockConfig.condition` controls visibility:

```typescript
{
  id: 'quorumThreshold',
  type: 'slider',
  title: 'Quorum Threshold (%)',
  condition: {
    field: 'gateType',       // ID of another sub-block in the same block
    value: 'threshold',      // show only when gateType === 'threshold'
  }
}
```

The condition can also use:
- `not: true` — invert the match
- `and: { field, value }` — additional AND condition
- **Function form** — `condition: (values) => ({ field, value })` for dynamic conditions

Evaluation happens in `lib/workflows/subblocks/visibility.ts` →
`evaluateSubBlockCondition()`, called both in the panel editor and on the canvas block
preview.

### Can sub-blocks have validation (e.g., "required field")?

**Yes — partially.** `SubBlockConfig.required` supports:
```typescript
required?: boolean
       | { field: string; value: ...; not?: boolean; and?: ... }  // conditional required
       | ((values) => { field, value, ... })                       // function form
```

`required: true` causes a visual missing-value indicator in the panel editor.
**However, there is no hard block on saving** — it is a UI hint only. Runtime enforcement
must be added in the block handler.

For governance blocks, add executor-level validation:
```typescript
// In GovernanceGateHandler.execute():
if (!inputs.voterList || inputs.voterList.length === 0) {
  throw new Error('GateConfig: voter list is required')
}
```

### What would we need for a gate config editor that shows 6 collapsible sections?

The existing `code` sub-block has `collapsible: true` / `defaultCollapsed: true`:
```typescript

  id: 'advancedPolicy',
  type: 'code',
  collapsible: true,
  defaultCollapsed: true,
}
```

For a **6-section gate config editor** with heterogeneous field types (not just code),
you have two options:

**Option A — 6 sub-blocks with `condition` show/hide + accordion headings:**
Use `condition: (values) => ...` to show/hide each section, plus `text` sub-blocks as
section headers. This works with zero new infrastructure.

**Option B — New `SubBlockType: 'accordion-sections'`:**
A richer compound component that renders N collapsible sections, each with their own
sub-fields. This would need:
1. New type in `SubBlockType` union (`blocks/types.ts`).
2. New React component in
   `panel/components/editor/components/sub-block/components/accordion-sections/`.
3. Dispatcher case added to the `SubBlock` component.

**Phase A recommendation:** Use Option A (existing types, `condition`-gated sections).
Option B is a Phase B polish task.

## 5. Block ID Stability

### How are block IDs generated? Are they stable across saves?

Block IDs are UUID v4 strings generated by `generateId()` from
`apps/sim/lib/core/utils/uuid.ts`:
```typescript
export function generateId(): string {
  return crypto.randomUUID()  // UUID v4
}
```

A block's ID is generated **once** when the block is first created (`batchAddBlocks()`
in the workflow store, or `handleAddOperation()` in the copilot edit-workflow handler).
The ID is stored in `BlockState.id`, persisted to `workflow_blocks.id` (primary key),
and loaded back unchanged on every page refresh. **The ID is stable for the lifetime of
the block in a given workflow.**

### Can we use the block ID as a `cp_id` for IGSL?

**Yes, with caveats:**

- **Within a workflow session:** The block ID is stable and unique — suitable as a CP
  identifier.
- **Across workflow copies/exports:** The `workflow.variables` JSON or a workflow export
  snapshot preserves IDs. But if the workflow is cloned via `preparePasteData()`, all
  block IDs are regenerated (see below).
- **Recommendation:** Use the block ID as the default `cp_id`. Add a `cp_id`
  sub-block field (`type: 'short-input', readOnly: true, defaultValue: generateId()`) if
  you need a stable identifier that survives across copies.

### What happens to the ID when you copy/paste a block?

`regenerateBlockIds()` in `apps/sim/stores/workflows/utils.ts` is called during paste
(`preparePasteData()` in the registry store):
```typescript
const newId = generateId()   // fresh UUID for every pasted block
blockIdMap.set(oldId, newId)
```

**Every pasted block gets a new UUID.** References inside sub-block values (like
`<blockName.output>` strings) are re-mapped to the new names via `nameMap`. Any
hardcoded `cp_id` sub-block field would keep its original value (since it is a user-set
value, not a computed field) — making it non-unique after paste.

**Governance implication:** Do not rely on `BlockState.id` as the canonical IGSL `cp_id`
across copies. Add a dedicated `cp_id` sub-block that defaults to a new UUID on block
creation and is not automatically remapped on paste.

## 6. Protocol-Level Settings

### Is there a concept of "workflow-level metadata" beyond the blocks?

**Yes — limited.** The `workflow` DB table (`packages/db/schema.ts`) has:
```
name, description, color, variables: json
```

The `WorkflowState` type (`stores/workflows/workflow/types.ts`) has:
```typescript
metadata?: {
  name?: string
  description?: string
  exportedAt?: string
}
variables?: Record<string, Variable>
```

`variables` is the main extensible field. Each `Variable` is:
```typescript
{ id, name, type: 'string'|'number'|'boolean'|'object'|'array'|'plain', value: unknown }
```

There is **no `jurisdiction`, `lifecycle`, or `protocolVersion` concept** in the current
schema.

### Where would protocol-level settings (name, jurisdiction, lifecycle) live?

Three viable options:

**Option A — Workflow `variables`:**  
Store `jurisdiction`, `lifecycle`, `protocolVersion` as typed workflow variables.
Zero schema changes. Accessible from any block via the variables store.
Downside: mixed with runtime variables; no dedicated UI.

**Option B — A "Protocol Metadata" block (special singleton block):**  
A `governance_protocol_header` block with `singleInstance: true`, `hideFromToolbar: false`,
and sub-blocks for `name`, `jurisdiction`, `lifecycle`, `protocolVersion`. The block is
inert at execution time (its handler just echoes inputs as outputs). The block is anchored
at the top of the canvas by convention.  
Pros: zero DB changes, consistent with existing block model, visible on canvas.  
Cons: users can accidentally delete it; no enforcement that it exists.

**Option C — Dedicated DB column / workflow settings panel:**  
Add a `protocolMetadata jsonb` column to the `workflow` table and a new settings panel
in the Deploy tab. Full DB migration required. Most correct long-term.

**Phase A recommendation:** Option B (Protocol Metadata block, `singleInstance: true`).
This is the fastest to implement and uses the existing `SubBlockConfig` system for UI.
Promote to Option C when the schema is stable.

### Could we use a special "Protocol" block or do we need a new panel/modal?

**A "Protocol" block is entirely sufficient for Phase A.** Pattern:

```typescript
export const GovernanceProtocolHeaderBlock: BlockConfig = {

  outputs: {
    protocolName: 'string',

  },

```

Downstream governance blocks reference `<Protocol.jurisdiction>` etc. via the standard
resolver syntax.

## 7. Execution Order

### How does Sim determine which block executes first?

The `ExecutionEngine` initializes a `readyQueue` in `initializeQueue()`
(`executor/execution/engine.ts`):

1. If `runFromBlockContext` → start from the specified block.
2. If `pendingBlocks` in context metadata (resume mode) → enqueue those.
3. If `triggerBlockId` provided → enqueue it.
4. Otherwise → find the `START_TRIGGER` / `STARTER` block in the DAG and enqueue it.

The DAG is built by `DAGBuilder` from the serialized workflow. `PathConstructor` finds
all blocks **reachable from the start/trigger block** — unreachable blocks are excluded
from execution entirely.

### Is execution guaranteed sequential in a chain A → B → C?

**Yes, for a simple linear chain.** The `EdgeManager` decrements `incomingEdges` after
each node completes. Block B is not enqueued until all its incoming edges have been
resolved (i.e., all upstream blocks that have edges to B have completed). So:

- A executes → `incomingEdges(B)` drops to 0 → B enqueues
- B executes → `incomingEdges(C)` drops to 0 → C enqueues

**Sequential within a chain is guaranteed.** Downstream blocks always wait for their
upstream blocks to finish.

### What about parallel branches? How does convergence work?

When a block has multiple *outgoing* edges (e.g., A→B and A→C), B and C are both
enqueued after A completes and run **concurrently** (`processQueue()` spawns multiple
`executeNodeAsync()` promises without awaiting each).

Convergence (e.g., B→D and C→D): D's `incomingEdges` starts at 2. After B completes
it drops to 1; after C completes it drops to 0 and D is enqueued. **D always waits for
all its upstream blocks.**

### Is upstream output available when downstream executes?

**Yes, guaranteed.** Before a downstream block runs, `BlockExecutor.execute()` calls
`resolver.resolveInputs()` which reads from `ctx.blockStates` (a `ReadonlyMap`). Block
states are populated via `ctx.setBlockOutput()` (or equivalent) after each block
completes. Since the downstream block is only enqueued after the upstream's
`handleNodeCompletion()` call (which happens after `state.setBlockOutput()`), the data
is always available.

**Governance implication:** Evidence produced by an upstream Evidence Collection block
is fully available when the downstream Gate block executes. No race conditions.

## 8. Custom Block Executor

### How does a block's execution handler work?

The handler system is in `executor/handlers/`. The `BlockHandler` interface
(`executor/types.ts`):

```typescript
export interface BlockHandler {
  canHandle(block: SerializedBlock): boolean

  execute(
    ctx: ExecutionContext,
    block: SerializedBlock,
    inputs: Record<string, any>
  ): Promise<BlockOutput | StreamingExecution>

  // Optional: used when the engine has node-level metadata (loop/parallel context)
  executeWithNode?: (
    ctx: ExecutionContext,
    block: SerializedBlock,
    inputs: Record<string, any>,
    nodeMetadata: {
      nodeId: string
      loopId?: string
      parallelId?: string
      branchIndex?: number
      branchTotal?: number
      originalBlockId?: string
      isLoopNode?: boolean
      executionOrder?: number
    }
  ) => Promise<BlockOutput | StreamingExecution>
}
```

`canHandle()` receives the `SerializedBlock` and returns `true` if the handler owns it.
Dispatch is first-match in the ordered handler list.

`inputs` are already resolved — all `<blockName.output>` references have been
substituted with actual values by `resolver.resolveInputs()` before the handler is
called.

### Can we register a custom executor for governance blocks?

**Yes — this is the standard extension point.** Steps:
1. Create `apps/sim/executor/handlers/governance/governance-gate-handler.ts`:
   ```typescript
   export class GovernanceGateHandler implements BlockHandler {
     canHandle(block: SerializedBlock): boolean {
       return block.metadata?.id === 'governance_gate'
     }
     async execute(ctx, block, inputs): Promise<BlockOutput> {
       // ... governance logic
     }
   }
   ```
2. Register it in `apps/sim/executor/handlers/registry.ts` **before**
   `GenericBlockHandler` (which is the fallback):
   ```typescript
   export function createBlockHandlers(): BlockHandler[] {
     return [
       new TriggerBlockHandler(),
       // ...
       new GovernanceGateHandler(),   // ← ADD BEFORE GenericBlockHandler
       new GenericBlockHandler(),
     ]
   }
   ```

No other changes required. The handler is instantiated fresh on every `createBlockHandlers()` call.

### What interface does a block handler implement?

```typescript
interface BlockHandler {
  canHandle(block: SerializedBlock): boolean
  execute(ctx, block, inputs): Promise<BlockOutput | StreamingExecution>
  executeWithNode?(ctx, block, inputs, nodeMetadata): Promise<...>  // optional
}
```

`BlockOutput` (from `apps/sim/blocks/types.ts`) is effectively `any` — the output
object is stored in `ctx.blockStates` and referenced by downstream blocks via
`<blockName.field>` syntax.

Key fields in `NormalizedBlockOutput` that the engine inspects:
- `_pauseMetadata: PauseMetadata` → triggers pause
- `error: string` → routes to error-handle edge
- `selectedOption: string` → activates a `condition-<id>` edge
- `selectedRoute: string` → activates a `router-<id>` edge

### How does the `human_in_the_loop` block pause and resume execution?

**Pause mechanism** (handler: `executor/handlers/human-in-the-loop/human-in-the-loop-handler.ts`):

1. Handler generates a unique `contextId` per pause point.
2. Builds `resumeLinks` (API URL + UI URL) using `buildResumeApiUrl()`.
3. Optionally sends notifications (Slack, email) with the resume link.
4. Returns `{ _pauseMetadata: { contextId, blockId, resumeLinks, ... } }`.
5. `ExecutionEngine.handleNodeCompletion()` detects `_pauseMetadata` → stores in
   `pausedBlocks` map → calls `serializePauseSnapshot()` to capture full engine state.
6. Engine returns `status: 'paused'` result. `pausedExecutions` DB table stores the
   snapshot.

**Resume mechanism** (`background/resume-execution.ts`):

1. Approver hits `POST /api/workflows/{wId}/execute/{execId}/resume/{contextId}` with
   form data.
2. `resume-execution.ts` job loads snapshot from `pausedExecutions`, reconstructs
   `ExecutionContext`, sets `metadata.resumeFromSnapshot = true`.
3. The resume input is injected into `ctx.blockStates` for the paused block.
4. Engine re-initializes from `pendingBlocks` (the blocks downstream of the paused block)
   and resumes.

**Key architectural insight for governance:** The current HITL model is **one pause →
one resume**. For multi-voter accumulation (pause N times, resume when quorum reached):

- **Option A (Phase A):** Chain N HITL blocks (one per voter) + a Condition block
  checking quorum. Works today, no engine changes. Can use a Loop with Variables to
  accumulate without N separate blocks.

- **Option B (Phase B):** A `GovernanceVoteHandler` that re-pauses on each voter
  response. Requires extending `handleNodeCompletion()` to support "re-pause after
  partial resume" — specifically, allowing a resumed block to emit `_pauseMetadata`
  again for the next voter.

The snapshot serializer (`executor/execution/snapshot-serializer.ts`) and the
`paused_executions` + `resume_queue` tables are fully reusable for either option.
